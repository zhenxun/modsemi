## 1. 專案概覽

**ModSemi** 是以企業級中後台場景為目標設計的 React 應用框架，核心目標是：

- 提供開箱即用的**四種佈局模式**（side / top / mix / double），讓開發者一行程式碼即可切換。
- 以**配置驅動**取代手動渲染，路由與選單共用同一份 `routesConfig`，消除同步維護的成本。
- 透過 **Zustand** 集中管理所有佈局設定，並自動持久化至 `localStorage`，刷新頁面不遺失設定。
- 整合 **Semi Design** 語意化 CSS 變數，讓亮色 / 暗色主題切換只需修改 `document.body` 的一個屬性。
- 系統啟動**預設為 Double 雙欄佈局**，首頁渲染精美的歡迎頁面。

### ⚠️ Semi Design 圖示套件命名注意

UI 元件套件名為 `@douyinfe/semi-ui-19`（帶 `-19` 代表 React 19 相容版本），但**圖示套件**在 npm 上的名稱是 `@douyinfe/semi-icons`，**沒有** `-19` 後綴。兩者為獨立套件，需分開安裝：


### 為何選擇 Modern.js 的嵌套路由？

Modern.js 的嵌套路由（Nested Routes）與 React Router v6 概念一致。`src/routes/layout.tsx` 是外層 Shell，`<Outlet />` 負責渲染對應路由的子頁面。這讓 ProLayout（佈局容器）只需掛載一次，子頁面切換時不會重新渲染佈局，大幅提升效能。

## 4. 框架約束說明（必讀）

### Modern.js 的固定命名規範

Modern.js 3.x 的檔案系統路由有以下**不可更改**的約束：

| 固定命名 | 說明 |
|----------|------|
| `src/routes/` | 框架掃描路由的根目錄，`source.entriesDir` 只能改父層，內部仍需 `routes/` 子目錄 |
| `layout.tsx` | 在任意路由層級作為「共享殼層」，包裹子路由的 `<Outlet />` |
| `page.tsx` | 代表該路由層級的實際頁面 |

**無法** 將 `src/routes/` 改為 `src/pages/`，也**無法** 將 `layout.tsx` 改為 `app.tsx`。

### 解決命名衝突的策略

早期 `src/config/routes.ts` 與 `src/routes/` 目錄並存，容易造成混淆。現行策略：

- `src/config/routes.ts` → **重命名為 `src/config/navigation.ts`**，在語意上明確區分「選單導航配置」與「框架路由目錄」
- `src/routes/page.tsx` 保持精簡，僅負責從 `src/pages/` 引入並 re-export 元件

### Less 支援配置

Modern.js 內建支援 Less，安裝 `less` 套件後只需在 `modern.config.ts` 加入 `tools.styleLoader` 即可啟用：

```typescript
// modern.config.ts
export default defineConfig({
  plugins: [appTools()],
  tools: {
    styleLoader: {},
  },
});
```

`.less` 檔案可直接在 TSX 元件中 import，無需任何額外 Webpack/Rspack 設定。

## 5. 核心模組詳解

### 5.1 導航配置 `config/navigation.ts`

> **此檔案原名為 `config/routes.ts`，已重命名以避免與 Modern.js 的 `src/routes/` 目錄混淆。**

這是整個系統的**唯一真實來源（Single Source of Truth）**，導航選單的結構完全由這份配置驅動。

#### `RouteItem` 型別定義

```typescript
interface RouteItem {
  text: string;           // 選單顯示文字
  itemKey: string;        // 唯一識別鍵，同時作為路由路徑
  icon?: ReactNode;       // 選單圖示（選填）
  children?: RouteItem[]; // 子選單（選填，最多支援兩層）
  hideInMenu?: boolean;   // 是否隱藏於導航列（true 時仍可正常路由，但不渲染到選單）
  path?: string;          // 自訂跳轉路徑（預設使用 itemKey）
}
```

#### `hideInMenu` 欄位的作用

設有 `hideInMenu: true` 的路由項目（如首頁 `"/"`）：
- **不會**出現在任何佈局的選單中
- **可以**正常被 React Router 渲染（URL 直接訪問有效）
- `useMenuData` 內部使用 `visibleRoutes = routesConfig.filter(item => !item.hideInMenu)` 統一過濾

#### 當前 `routesConfig` 範例

```typescript
export const routesConfig: RouteItem[] = [
  {
    text: '歡迎',
    itemKey: '/',
    hideInMenu: true,          // 首頁不出現在選單中
  },
  {
    text: '儀表板',
    itemKey: '/dashboard',
    children: [
      { text: '工作台', itemKey: '/dashboard/workplace' },
      { text: '分析頁', itemKey: '/dashboard/analysis' },
      { text: '監控頁', itemKey: '/dashboard/monitor' },
    ],
  },
  // ... 其他模組
];
```

#### 如何新增選單項

```typescript
// 在 routesConfig 陣列中新增：
{
  text: '報表中心',
  itemKey: '/report',
  children: [
    { text: '銷售報表', itemKey: '/report/sales' },
    { text: '財務報表', itemKey: '/report/finance' },
  ],
},
```

新增後，對應的路由頁面需建立於 `src/routes/report/sales/page.tsx`（Modern.js 檔案系統路由規則），實際元件放在 `src/pages/Report/Sales/index.tsx`，`page.tsx` 只需 re-export。

### 5.2 狀態管理 `store/layoutStore.ts`

#### 狀態結構

```typescript
interface LayoutState {
  layoutMode: 'side' | 'top' | 'mix' | 'double'; // 佈局模式，預設 'double'
  themeMode: 'light' | 'dark';                    // 主題模式
  fixedHeader: boolean;                           // 是否固定頂欄
  showBreadcrumb: boolean;                        // 是否顯示麵包屑
  settingDrawerOpen: boolean;                     // 設定面板開關（不持久化）
  doubleFirstKey: string;                         // double 模式第一欄選中項
}
```

#### 預設值

```typescript
const initialState: LayoutState = {
  layoutMode: 'double',    // ← 系統啟動預設為雙欄佈局
  themeMode: 'light',
  fixedHeader: true,
  showBreadcrumb: true,
  settingDrawerOpen: false,
  doubleFirstKey: '/dashboard',
};
```

#### Zustand `persist` 持久化

```typescript
persist(
  (set) => ({ ... }),
  {
    name: 'modsemi-layout',
    partialize: (state: LayoutState & LayoutActions) => ({
      // 僅持久化使用者偏好設定
      layoutMode: state.layoutMode,
      themeMode: state.themeMode,
      fixedHeader: state.fixedHeader,
      showBreadcrumb: state.showBreadcrumb,
      doubleFirstKey: state.doubleFirstKey,
      // settingDrawerOpen ← 刻意不持久化，刷新後抽屜應保持關閉
    }),
  }
)
```

> **⚠️ API 注意**：Zustand 5.x 中正確的欄位名為 **`partialize`**（不是 `partializer`）。函數參數也須明確標注型別 `state: LayoutState & LayoutActions`，否則 TypeScript 會報 `any` 推斷錯誤。

#### 在元件中使用

```typescript
// ✅ 推薦：只訂閱需要的欄位，最小化重新渲染
const layoutMode = useLayoutStore(s => s.layoutMode);
const setLayoutMode = useLayoutStore(s => s.setLayoutMode);

// ⚠️ 不推薦：訂閱整個 store，任何欄位變更都觸發重新渲染
const store = useLayoutStore();
```

### 5.3 導航數據 Hook `hooks/useMenuData.ts`

`useMenuData` 是系統最核心的計算層，根據 `layoutMode`、`pathname`、`doubleFirstKey` 自動推導出所有佈局元件需要的導航數據。

#### `visibleRoutes` 過濾機制

```typescript
// 模組層級的靜態過濾，只執行一次，不需放在 useMemo 內
const visibleRoutes = routesConfig.filter(item => !item.hideInMenu);
```

所有選單計算（`findFirstLevelKey`、`computeSelectedAndOpen`、`buildMenuData`）均基於 `visibleRoutes`，而非原始的 `routesConfig`，確保 `hideInMenu: true` 的項目（如歡迎頁 `"/"`）永遠不會出現在導航列中。

#### 返回值說明

```typescript
interface BreadcrumbItem {
  path: string;  // 路由路徑（用於點擊導航）
  text: string;  // 顯示文字（來自 routesConfig.text）
}

interface MenuDataResult {
  firstLevelMenus: RouteItem[];    // 一級選單列表（已過濾 hideInMenu）
  secondLevelMenus: RouteItem[];   // 二級選單（mix/double 模式專用）
  selectedKeys: string[];          // 當前高亮的選單 key
  openKeys: string[];              // 側邊欄展開的父節點 key
  activeFirstKey: string;          // 一級選中的 key（mix/double 定位用）
  breadcrumbs: BreadcrumbItem[];   // 麵包屑資料，從 routesConfig 查詢文字
}
```

#### `breadcrumbs` 計算規則

`buildBreadcrumbs(pathname)` 搜尋**完整** `routesConfig`（包含 `hideInMenu` 項目），確保每個路徑都能找到正確的文字顯示：

| pathname | breadcrumbs |
|----------|-------------|
| `/` | `[{ path: '/', text: '歡迎' }]` |
| `/dashboard/workplace` | `[{ path: '/dashboard', text: '儀表板' }, { path: '/dashboard/workplace', text: '工作台' }]` |
| `/unknown` | `[{ path: '/unknown', text: '/unknown' }]`（回退：顯示路徑本身）|

```typescript
// buildBreadcrumbs 核心邏輯
function buildBreadcrumbs(pathname: string): BreadcrumbItem[] {
  for (const item of routesConfig) {
    if (pathname === item.itemKey) {
      return [{ path: item.itemKey, text: item.text }];
    }
    if (item.children) {
      for (const child of item.children) {
        if (pathname === child.itemKey || pathname.startsWith(`${child.itemKey}/`)) {
          return [
            { path: item.itemKey, text: item.text },
            { path: child.itemKey, text: child.text },
          ];
        }
      }
    }
  }
  return [{ path: pathname, text: pathname }];
}
```

#### 各佈局模式下的返回差異

| 模式 | `firstLevelMenus` | `secondLevelMenus` | 備註 |
|------|-------------------|--------------------|------|
| `side` | 完整 `visibleRoutes` 樹 | `[]` | Nav 自行處理巢狀展開 |
| `top` | 完整 `visibleRoutes` 樹 | `[]` | 水平選單，子項以 dropdown 展示 |
| `mix` | `visibleRoutes` 一級項 | 當前一級項的 children | 頂部父節點 + 側邊子項 |
| `double` | `visibleRoutes`（圖示欄用） | 由 `doubleFirstKey` 決定 | 圖示欄點擊更新二級 |

#### 路徑匹配演算法

`findFirstLevelKey(pathname)` 使用前綴匹配策略：

1. 精確匹配 `visibleRoutes` 一級節點的 `itemKey`
2. 精確匹配所有二級節點的 `itemKey`，返回其父節點 key
3. 均未匹配時，回退至 `visibleRoutes[0].itemKey`

這確保訪問 `/dashboard/analysis` 時，`/dashboard` 節點正確高亮。

```tsx
};
```

```
src/config/global.tsx
```

### 5.10 麵包屑系統

麵包屑系統由三個層次組成，職責清晰分離：

```
useMenuData()           ← 計算 breadcrumbs 資料（純資料層）
    ↓
LayoutBreadcrumb.tsx    ← 同步 document.title + 決定是否顯示（控制層）
    ↓
globalConfig.breadcrumbComponent  ← 實際渲染 UI（呈現層，可替換）
```

#### `AppBreadcrumb`（預設麵包屑元件）

位於 `src/components/AppBreadcrumb/index.tsx`，使用 Semi Design 的 `Breadcrumb.Item` 子元素方式渲染：

```tsx
export function AppBreadcrumb({ breadcrumbs }: BreadcrumbComponentProps) {
  const navigate = useNavigate();
  const lastIndex = breadcrumbs.length - 1;

  return (
    <Breadcrumb>
      {breadcrumbs.map((b, i) => {
        const isLast = i === lastIndex;
        return (
          <Breadcrumb.Item
            key={b.path}
            onClick={isLast ? undefined : () => navigate(b.path)}
            style={isLast ? undefined : { cursor: 'pointer' }}
          >
            {b.text}
          </Breadcrumb.Item>
        );
      })}
    </Breadcrumb>
  );
}
```

**點擊行為**：

| 項目 | 行為 |
|------|------|
| 非最後一項（父層） | 可點擊，`navigate(path)` 跳轉 |
| 最後一項（當前頁） | 不可點擊（慣例：當前頁點自己無意義） |

> **⚠️ 為何不用 `routes` prop**：Semi Design 的 `Breadcrumb` 使用 `routes` prop 時，預設 `itemRender` 顯示的是 `path` 字串（作為超連結文字），而非 `breadcrumbName`。正確做法是使用 `Breadcrumb.Item` 子元素方式，直接控制顯示內容。

#### `LayoutBreadcrumb`（佈局層包裝）

位於 `src/layouts/ProLayout/LayoutBreadcrumb.tsx`，是四個佈局共用的麵包屑掛載點：

```tsx
export function LayoutBreadcrumb({ breadcrumbs, show }: LayoutBreadcrumbProps) {
  const customTitle = usePageTitleStore(s => s.customTitle);

  // 自動同步 document.title（見 5.11）
  useEffect(() => {
    const autoTitle = breadcrumbs[breadcrumbs.length - 1]?.text;
    document.title = customTitle ?? autoTitle ?? document.title;
  }, [breadcrumbs, customTitle]);

  if (!show || breadcrumbs.length === 0) return null;

  const BreadcrumbComp = globalConfig.breadcrumbComponent;
  return (
    <div style={{ marginBottom: 16 }}>
      <BreadcrumbComp breadcrumbs={breadcrumbs} />
    </div>
  );
}
```

`show` 對應 `layoutStore.showBreadcrumb`，使用者可在設定面板切換。

#### 在四個佈局中的使用方式

所有佈局的 `Content` 區開頭使用同一行：

```tsx
<LayoutBreadcrumb breadcrumbs={breadcrumbs} show={showBreadcrumb} />
```

### 5.11 頁面標題同步 `usePageTitle`

#### 自動模式

每次路由切換，`LayoutBreadcrumb` 的 `useEffect` 自動執行：

```
路由切換 → breadcrumbs 更新 → document.title = 最後一個 breadcrumb 的 text
```

例如進入 `/dashboard/workplace`：`document.title = '工作台'`

#### 開發者覆寫模式

在頁面元件中呼叫 `usePageTitle(title)` 即可強制覆寫：

```tsx
// src/pages/Dashboard/Workplace/index.tsx
import { usePageTitle } from '../../hooks/usePageTitle';

export default function Workplace() {
  usePageTitle('工作台 — 即時總覽');  // ← 只需這一行
  return <div>...</div>;
}
```

`document.title` 將變為 `'工作台 — 即時總覽'`，而非自動推導的 `'工作台'`。

#### 底層機制

**`src/store/pageTitleStore.ts`**：

```typescript
interface PageTitleState {
  customTitle: string | null;  // null 代表使用自動推導
  setCustomTitle: (title: string | null) => void;
}
```

**`src/hooks/usePageTitle.ts`**：

```typescript
export function usePageTitle(title: string) {
  const setCustomTitle = usePageTitleStore(s => s.setCustomTitle);

  useEffect(() => {
    setCustomTitle(title);
    return () => setCustomTitle(null);  // 離開頁面自動清除
  }, [title, setCustomTitle]);
}
```

#### 為什麼覆寫一定會贏（執行時序保證）

React 的 `useEffect` 按照**從子到父**的順序執行（children first）。頁面元件是 `<Outlet />` 的子元素，比 `LayoutBreadcrumb` 層級更深：

```
Commit 階段 useEffect 執行順序：

1. 頁面元件（子）：setCustomTitle('工作台 — 即時總覽')
   └─ Zustand store 同步更新：customTitle = '工作台 — 即時總覽'

2. LayoutBreadcrumb（父）：
   └─ 讀取 customTitle → 使用 '工作台 — 即時總覽' 設定 document.title
```

離開頁面時，頁面的 `useEffect` cleanup 先執行 `setCustomTitle(null)`，`LayoutBreadcrumb` 後執行，回退至自動推導模式。

#### 標題優先級

```
usePageTitle('自訂')  >  breadcrumbs[-1].text  >  維持當前 document.title
```

**適用場景**：功能模組中等（5–8 個），每個模組下有多個子頁面，常見於大型 SaaS 後台。

**實作要點**：
- 頂欄只展示**一級父節點**，點擊後自動跳轉至該模組的第一個子頁面
- 側欄顯示**當前一級模組的二級子項**，切換一級時同步更新
- 當一級節點無 `children` 時，側欄自動隱藏（`sideItems.length > 0` 條件判斷）

### 6.4 Double（雙欄導航）

```
┌──────────────────────────────────────────────┐
│  Logo                                  [⚙️]  │  ← Header
├────┬─────────────────┬─────────────────────  │
│ 📊 │  工作台          │                       │
│ 📝 │  分析頁          │    [Content]          │
│ 📋 │  監控頁          │                       │
│ ⚙️ │                  │                       │
└────┴─────────────────┴───────────────────────┘
  ↑           ↑
第一欄      第二欄
(56px)    (文字選單)
```

**適用場景**：功能極豐富（8 個模組以上）的超大型後台，如 IDE、設計工具、企業 ERP。

**系統啟動預設為此模式。**

**實作要點**：

**第一欄（圖示欄，56px）**：
- 使用原生 `<button>` 元素（非 `Nav`），精確控制樣式與無障礙
- 點擊呼叫 `setDoubleFirstKey(item.itemKey)`，更新 Zustand 狀態
- 高亮背景用 `var(--semi-color-primary-light-default)` 實現
- `Tooltip` 提示完整文字名稱
- 若無 `icon`，自動取 `text.slice(0, 1)` 作回退顯示

**第二欄（文字欄）**：
- 根據 `doubleFirstKey` 從 `visibleRoutes` 找對應父節點，取其 `children` 渲染
- `doubleFirstKey` 持久化至 `localStorage`，刷新後維持上次選中模組

**與 mix 模式的差異**：

| 面向 | Mix | Double |
|------|-----|--------|
| 一級選單位置 | 頂部水平 | 左側 56px 圖示欄 |
| 二級選單位置 | 左側 Sider | 中間 Sider |
| 一級選中觸發 | 路徑自動計算 | 點擊圖示手動更新 `doubleFirstKey` |
| 適合模組數量 | 5–8 個 | 8 個以上 |

```
     │
     ▼
useLocation().pathname               ← 取得新路徑
     │
     ▼
useMenuData()                        ← 重新計算導航數據
  ├── visibleRoutes                  ← 過濾 hideInMenu 項目
  ├── findFirstLevelKey()            ← 前綴匹配找一級父節點
  ├── computeSelectedAndOpen()       ← 計算高亮 & 展開 keys
  └── buildMenuData()                ← 根據 layoutMode 分派
     │
     ▼
各佈局元件接收新數據
  ├── selectedKeys  → Nav 高亮
  ├── openKeys      → Nav 展開
  └── secondLevelMenus → Sider 更新（mix/double）
```

### 設定變更流程

```
使用者開啟 SettingDrawer 切換佈局模式
     │
     ▼
setLayoutMode(newMode)               ← Zustand Action
     │
     ├─→ layoutStore 記憶體狀態更新
     │      └─→ persist 中介層同步寫入 localStorage
     │
     └─→ useLayoutStore() 訂閱者重新渲染
            └─→ ProLayout/index.tsx 依新 mode 渲染子元件
```

### 9.7 動態路由權限過濾

在 `useMenuData.ts` 中注入權限過濾邏輯，替換 `visibleRoutes`：

```typescript
import { useAuthStore } from '../store/authStore';

export function useMenuData(): MenuDataResult {
  const { permissions } = useAuthStore();

  const visibleRoutes = useMemo(
    () => filterRoutes(routesConfig, permissions),
    [permissions]
  );

  // 其餘邏輯不變，使用上方的 visibleRoutes
}

function filterRoutes(items: RouteItem[], permissions: string[]): RouteItem[] {
  return items
    .filter(item => !item.hideInMenu)
    .filter(item => !item.permission || permissions.includes(item.permission))
    .map(item => ({
      ...item,
      children: item.children
        ? filterRoutes(item.children, permissions)
        : undefined,
    }));
}
```

## 11. 常見問題 FAQ

### Q1：切換佈局後為何部分樣式沒有即時更新？

**A**：ProLayout 根據 `layoutMode` 條件渲染子元件，React 會卸載舊元件並掛載新元件，樣式應完全重置。若出現殘留，請確認沒有在 `global.less` 中使用全域 CSS class 覆蓋 Semi 的選擇器。

---

### Q2：`double` 模式下第一欄沒有顯示圖示？

**A**：`routesConfig` 中的 `icon` 欄位是選填的。若未提供，系統自動以 `item.text.slice(0, 1)` 作回退。若要使用 Semi 圖示，在 `navigation.ts` 中引入並設定（注意正確的套件名）：

```typescript
import { IconHome } from '@douyinfe/semi-icons';   // ← 正確套件名（無 -19）

{
  text: '儀表板',
  itemKey: '/dashboard',
  icon: <IconHome />,
  children: [...],
}
```

---

### Q3：如何讓某個頁面不出現在選單中，但 URL 仍可正常訪問？

**A**：在 `navigation.ts` 的對應路由項設定 `hideInMenu: true`，`useMenuData` 內部使用 `visibleRoutes` 會自動過濾這些項目，但 React Router 仍能正常渲染該路徑：

```typescript
{ text: '訂單詳情', itemKey: '/order/detail', hideInMenu: true }
```

---

### Q4：Zustand `persist` 的狀態在不同瀏覽器分頁間會同步嗎？

**A**：不會即時同步。`persist` 使用 `localStorage`，多個分頁共享儲存層，但各分頁的記憶體狀態獨立。若需跨分頁即時同步，需搭配 `BroadcastChannel` API 或 Zustand 的 `shared` 中介層。

---

### Q5：為什麼 `src/routes/` 不能改名為 `src/pages/`？

**A**：這是 Modern.js 3.x 檔案系統路由的硬性約束。框架在 `source.entriesDir`（預設 `src`）下固定尋找名為 `routes` 的子目錄作為路由根目錄，無法透過設定更改此名稱。  

現行解決方案：`src/routes/` 只放框架需要的薄包裝檔（`layout.tsx`、`page.tsx`），實際業務元件全部放在可自由命名的 `src/pages/` 中。

---

### Q6：麵包屑顯示的文字怎麼決定的？能不能自訂？

**A**：麵包屑文字完全來自 `src/config/navigation.tsx` 的 `routesConfig[].text` 欄位，由 `useMenuData` 的 `buildBreadcrumbs()` 函式自動查詢。要改文字，只需修改 `routesConfig` 對應項目的 `text` 即可，所有佈局同步更新。

如需自訂渲染方式（顏色、分隔符、圖示等），在 `src/config/global.tsx` 替換 `breadcrumbComponent` 欄位（見 9.4）。

---

### Q7：麵包屑點擊父層後跳到的頁面是 `/dashboard` 還是 `/dashboard/workplace`？

**A**：`AppBreadcrumb`（預設實作）點擊非最後一項時，直接 `navigate(b.path)`——父層項目的 `path` 就是 `routesConfig` 的 `itemKey`，例如 `/dashboard`。

`/dashboard` 本身通常沒有對應的 `page.tsx`，Modern.js 會渲染 404 頁面或顯示空白。若需要點擊父層跳轉至第一個子頁，可在自訂 `breadcrumbComponent` 中實作此邏輯：

```tsx
const target = breadcrumbs[i].path;
const parentRoute = routesConfig.find(r => r.itemKey === target);
const dest = parentRoute?.children?.[0]?.itemKey ?? target;
navigate(dest);
```

---

### Q8：`usePageTitle` 與麵包屑自動 title 誰的優先級高？

**A**：`usePageTitle` 優先。呼叫 `usePageTitle('自訂標題')` 後，`pageTitleStore.customTitle` 被設為非 null 值，`LayoutBreadcrumb` 的 `useEffect` 看到 `customTitle` 非空，直接使用自訂標題，忽略 breadcrumbs 推導結果。

離開頁面時，`usePageTitle` 的 cleanup 函數自動執行 `setCustomTitle(null)`，下個頁面恢復自動推導模式。

---

### Q9：如何確認 404 頁面有正確運作？

**A**：在瀏覽器中直接訪問任何不存在的路徑（如 `/this-does-not-exist`），應看到 404 頁面。若未顯示，請確認 `src/routes/$.tsx` 已建立，且其 `export default` 函數正常回傳元件。

---

### Q10：為何選擇 Zustand 而非 Redux 或 Context API？

**A**：
- **vs Redux**：Redux 需要大量樣板程式碼（actions / reducers / selectors），對佈局設定這種簡單狀態來說過度設計。
- **vs Context API**：Context 在任何欄位變更時都會導致所有消費者重新渲染。Zustand 支援選擇性訂閱 `useLayoutStore(s => s.layoutMode)`，只有訂閱的欄位變更才觸發渲染。
- **持久化**：`persist` 中介層讓 `localStorage` 同步只需數行設定。

### 13.2 `authStore` — 認證狀態

**檔案：** `src/store/authStore.ts`

#### 型別定義

```typescript

```

### 13.3 登入頁與路由守衛

**登入頁路由：** `/login`（`src/routes/login/page.tsx`）

#### 路由守衛機制

路由守衛位於 `src/routes/layout.tsx`，採用**宣告式重導**（`<Navigate>`），而非 `useEffect`，避免非同步競爭問題：

```tsx
// src/routes/layout.tsx
export default function Layout() {
  const { isLoggedIn } = useAuthStore();
  const isLoginPage = location.pathname === '/login';

  // 未登入 → 直接重導向，不渲染任何內容
  if (!isLoggedIn && !isLoginPage) {
    return <Navigate to="/login" replace />;
  }

  // 登入頁不套 ProLayout
  if (isLoginPage) {
    return <Outlet />;
  }

  // 已登入 → 正常渲染佈局
  return (
    <>
      <ProLayout title="ModSemi" logo={logo}>
        <Outlet />
      </ProLayout>
      <SettingDrawer />
    </>
  );
}
```

#### 登入流程

```
使用者輸入帳密
      ↓
authStore.login(username, password)
      ↓ 驗證成功（回傳 true）
isLoggedIn = true 寫入 localStorage
      ↓
navigate('/', { replace: true })
      ↓
layout.tsx 重渲染 → 不命中 <Navigate> → 渲染 ProLayout
      ↓
顯示歡迎頁（/）
```

#### 登出流程

```
UserAvatar 點擊「登出」
      ↓
authStore.logout()（清除 currentUser, isLoggedIn = false）
      ↓
navigate('/login', { replace: true })
      ↓
layout.tsx 重渲染 → 命中 <Navigate to="/login"> → 顯示登入頁
```

### 13.4 `useAccess` — 權限 Hook

**檔案：** `src/hooks/useAccess.ts`

#### 設計理念

「角色 → 權限」的對映邏輯**集中於此 Hook**，業務程式碼只消費語意化 key（如 `isAdmin`、`canViewSettings`），不直接判斷 roles 字串。這樣的好處是：

- 修改角色邏輯只需改一個檔案
- 業務程式碼與角色命名解耦
- 方便撰寫單元測試（mock hook 即可）

#### 目前權限定義

```typescript
interface AccessConfig {
  isAdmin: boolean;         // roles 包含 'admin'
  canViewDashboard: boolean;  // 所有登入使用者
  canViewForm: boolean;       // 所有登入使用者
  canViewList: boolean;       // 所有登入使用者
  canViewDetail: boolean;     // 所有登入使用者
  canViewSettings: boolean;   // 僅 isAdmin
}
```

#### 使用方式

```tsx
import { useAccess } from '../hooks/useAccess';

function SomePage() {
  const access = useAccess();

  return (
    <div>
      {/* 只有管理員才顯示這個按鈕 */}
      {access.isAdmin && <AdminButton />}

      {/* 有查看設定權限才顯示設定連結 */}
      {access.canViewSettings && <Link to="/settings">系統設定</Link>}
    </div>
  );
}
```

### 13.5 `Access` 組件 — 條件渲染

**檔案：** `src/components/Access/index.tsx`

比 `{condition && <Component />}` 更語意化的包裝組件，支援 `fallback` 替代內容。

#### Props

| Prop | 型別 | 必填 | 說明 |
|------|------|------|------|
| `accessible` | `boolean` | ✅ | 是否有存取權限 |
| `fallback` | `ReactNode` | ❌ | 無權限時顯示（預設 `null`） |
| `children` | `ReactNode` | ✅ | 有權限時渲染的內容 |

#### 使用範例

```tsx
import { Access } from '../components/Access';
import { useAccess } from '../hooks/useAccess';

function SettingsPage() {
  const access = useAccess();

  return (
    <div>
      <h1>設定頁面</h1>

      {/* 基本用法：無權限時什麼都不顯示 */}
      <Access accessible={access.isAdmin}>
        <DangerZonePanel />
      </Access>

      {/* 搭配 fallback：無權限時顯示提示 */}
      <Access
        accessible={access.canViewSettings}
        fallback={<p>您沒有查看此內容的權限。</p>}
      >
        <SystemConfigPanel />
      </Access>
    </div>
  );
}
```

#### 與直接判斷的比較

```tsx
// ❌ 不推薦：散落各處的 roles 判斷
const roles = useAuthStore(s => s.currentUser?.roles ?? []);
{roles.includes('admin') && <DeleteButton />}

// ✅ 推薦：語意化、統一管理
const access = useAccess();
<Access accessible={access.isAdmin}>
  <DeleteButton />
</Access>
```

### 13.6 導航權限控制

透過在 `routesConfig` 的路由項目設定 `access` 欄位，即可讓 `useMenuData` 在計算選單時**自動過濾無權限的項目**，使用者不會看到也無法點選。

#### 欄位定義

```typescript
// src/config/navigation.tsx
export interface RouteItem {
  text: string;
  itemKey: string;
  icon?: ReactNode;
  children?: RouteItem[];
  hideInMenu?: boolean;
  path?: string;
  /**
   * 存取權限 key，對應 AccessConfig 中的欄位名稱。
   * 設定後，使用者須擁有對應權限才能在選單中看到此項目。
   * 未設定則所有登入使用者均可見。
   */
  access?: string;
}
```

#### 設定範例

```typescript
// src/config/navigation.tsx
export const routesConfig: RouteItem[] = [
  {
    text: '儀表板',
    itemKey: '/dashboard',
    icon: <IconBarChartVStroked />,
    // 未設定 access → 所有登入使用者可見
    children: [
      { text: '工作台', itemKey: '/dashboard/workplace' },
    ],
  },
  {
    text: '系統設定',
    itemKey: '/settings',
    icon: <IconSettingStroked />,
    access: 'canViewSettings',   // ← 只有 canViewSettings = true 的使用者才看得到
    children: [
      { text: '帳號設定', itemKey: '/settings/account' },
      { text: '權限管理', itemKey: '/settings/permission' },
      // 子項目也可以個別設定 access
      {
        text: '進階設定',
        itemKey: '/settings/advanced',
        access: 'isAdmin',       // ← 子項目獨立設定
      },
    ],
  },
];
```

#### 過濾邏輯

`useMenuData` 內部呼叫 `useAccess()` 後，對路由做兩層過濾：

```
routesConfig
  └─ 過濾 hideInMenu = true          （隱藏於選單，如歡迎頁）
  └─ 過濾 access 不符當前使用者      （無權限項目自動消失）
      └─ 對每個一級項目的 children
         再做一次 access 過濾         （子項目也獨立控制）
```

> **💡 提示：** 導航權限控制只影響選單的顯示，不保護路由本身。若要防止使用者直接輸入 URL 存取，需在頁面元件或路由層加上額外保護（例如在頁面內呼叫 `useAccess` 並返回 403 提示）。

### 13.7 擴充角色與權限

#### Step 1：新增角色

在 `authStore.ts` 的 `MOCK_USERS` 或後端 API 回傳的使用者資料中，加入新角色：

```typescript
{ name: 'Editor', roles: ['editor'] }
```

#### Step 2：在 `useAccess` 中定義對應權限

```typescript
// src/hooks/useAccess.ts
export interface AccessConfig {
  isAdmin: boolean;
  isEditor: boolean;          // 新增 editor 角色
  canViewDashboard: boolean;
  canViewForm: boolean;
  canEditContent: boolean;    // 新增對應能力
  canPublish: boolean;
  canViewSettings: boolean;
}

export function useAccess(): AccessConfig {
  const roles = currentUser?.roles ?? [];
  const isAdmin = roles.includes('admin');
  const isEditor = roles.includes('editor');

  return {
    isAdmin,
    isEditor,
    canViewDashboard: true,
    canViewForm: isAdmin || isEditor,  // 管理員和編輯者都可以
    canEditContent: isAdmin || isEditor,
    canPublish: isAdmin,               // 只有管理員可以發布
    canViewSettings: isAdmin,
  };
}
```

#### Step 3：在 navigation 中使用新權限

```typescript
{
  text: '內容管理',
  itemKey: '/content',
  access: 'canEditContent',   // ← 使用新定義的 key
  children: [
    { text: '文章列表', itemKey: '/content/articles' },
    {
      text: '發布管理',
      itemKey: '/content/publish',
      access: 'canPublish',   // ← 子項目更細粒度控制
    },
  ],
}
```

#### Step 4：在頁面元件中使用

```tsx
import { Access } from '../components/Access';
import { useAccess } from '../hooks/useAccess';

export default function ArticleEditor() {
  const access = useAccess();

  return (
    <div>
      <ArticleForm />

      <Access accessible={access.canPublish}>
        <PublishButton />
      </Access>

      <Access
        accessible={access.isAdmin}
        fallback={<p>只有管理員才能刪除文章。</p>}
      >
        <DeleteButton />
      </Access>
    </div>
  );
}
```

## 14. API 程式碼自動生成（Orval）

本章說明 ModSemi 如何整合 **[Orval](https://orval.dev/)** 實現「給一份 Swagger JSON → 自動產出 TypeScript API 請求程式碼」的完整流程。

---

### 14.1 概覽與架構

#### 什麼是 Orval？

Orval 是一套 TypeScript 程式碼生成工具，它能讀取符合 **OpenAPI v3 / Swagger v2** 規格的 JSON 或 YAML 檔案，自動產出：

- **TypeScript 型別定義**：所有 request body、response、query params 的 interface / type
- **API 請求函式**：使用 axios、fetch 或 React Query hooks 等多種客戶端
- **（可選）MSW Mock handlers**：不需後端即可進行前端開發與測試

#### ModSemi 整合架構

```
┌─────────────────────────────────────────────────────────┐
│                  Swagger / OpenAPI Spec                  │
│              swagger/sample.json (你的 spec)              │
└──────────────────────────┬──────────────────────────────┘
                           │  pnpm api:gen
                           ▼
                    orval.config.ts
                    （讀取設定，驅動生成）
                           │
          ┌────────────────┴────────────────┐
          ▼                                 ▼
 src/api/generated/                src/api/generated/model/
 ├── user.ts  (User API 函式)       ├── user.ts
 └── article.ts (Article API 函式)  ├── userListResponse.ts
                                    ├── createUserRequest.ts
                                    └── index.ts (re-export)
                           │
                           ▼
                  src/api/instance.ts
          （所有 API 函式共用的 axios instance）
          ├── baseURL 設定
          ├── 請求攔截：自動注入 Bearer Token
          └── 回應攔截：401 跳轉、500 記錄錯誤
```

#### 技術選型說明

| 選項 | 選擇 | 原因 |
|------|------|------|
| HTTP 客戶端 | `axios` | 攔截器豐富、社群龐大、易於整合 Auth Token |
| 生成模式 | `'axios'`（工廠函式） | 可注入自訂 instance，彈性高 |
| 輸出模式 | `'tags'` | 依 OpenAPI tags 拆分檔案，大型 spec 亦清晰 |
| Mutator | `customInstance` | 所有生成函式都走同一個已設定好的 axios instance |

---

### 14.2 快速上手：三步驟生成 API

#### 步驟一：放入 Swagger spec

將後端提供的 Swagger / OpenAPI JSON 放到 `swagger/` 目錄下：

```
swagger/
└── your-api.json   ← 放在這裡
```

#### 步驟二：更新 orval.config.ts

打開 `orval.config.ts`，修改 `input.target` 指向你的 spec 檔案：

```typescript
export default defineConfig({
  yourApi: {                                   // 任意命名
    input: {
      target: './swagger/your-api.json',       // 改這裡
    },
    output: {
      mode: 'tags',
      target: './src/api/generated',
      schemas: './src/api/generated/model',
      client: 'axios',
      override: {
        mutator: {
          path: './src/api/instance.ts',
          name: 'customInstance',
        },
      },
    },
  },
});
```

#### 步驟三：執行生成指令

```bash
pnpm api:gen
```

生成完成後，`src/api/generated/` 目錄下即出現依 tag 分類的 TypeScript 檔案，可直接 import 使用。

### 14.4 Axios Instance 詳解

**檔案**：`src/api/instance.ts`

此檔案是整個 API 層的核心，所有生成的請求函式都透過 `customInstance` 發送請求。

#### 主要設定

```typescript
export const apiInstance = axios.create({
  baseURL: BASE_URL,  // 預設 '/api'，可透過環境變數覆蓋
  timeout: 15000,     // 15 秒逾時
  headers: {
    'Content-Type': 'application/json',
  },
});
```

#### 請求攔截器（自動注入 Token）

```typescript
apiInstance.interceptors.request.use(config => {
  // 從 Zustand persist 的 localStorage 讀取 token
  const raw = localStorage.getItem('modsemi-auth');
  if (raw) {
    const state = JSON.parse(raw);
    const token = state?.state?.token;
    if (token) {
      config.headers.Authorization = `Bearer ${token}`;
    }
  }
  return config;
});
```

> 當 `authStore` 加入 `token` 欄位後，此攔截器會自動生效，無需修改任何 API 呼叫點。

#### 回應攔截器（統一錯誤處理）

| HTTP 狀態碼 | 處理行為 |
|------------|---------|
| `401` | 清除 `modsemi-auth` localStorage，導向 `/login` |
| `403` | `console.warn` 記錄權限不足 |
| `>= 500` | `console.error` 記錄伺服器錯誤 |
| 網路錯誤 | `console.error` 記錄連線問題 |

#### `customInstance` — Orval Mutator

```typescript
export const customInstance = <T>(config: AxiosRequestConfig): Promise<T> => {
  return apiInstance(config).then(response => response.data as T);
};
```

這個函式是連接 Orval 生成程式碼與 axios instance 的橋樑：

- Orval 生成的每個 API 函式都呼叫 `customInstance()`
- `customInstance` 使用已設定好攔截器的 `apiInstance` 發送請求
- 自動解包 `response.data`，讓呼叫端直接取得業務資料

### 14.5 orval.config.ts 設定說明

#### `input` — 輸入設定

```typescript
input: {
  // 支援本地路徑或遠端 URL
  target: './swagger/sample.json',
  // target: 'https://api.example.com/v3/api-docs',
}
```

#### `output.mode` — 輸出模式

| 模式 | 說明 | 適合場景 |
|------|------|---------|
| `'single'` | 所有 API 合併到一個檔案 | 小型 spec（< 20 個 endpoints） |
| `'split'` | models 與 functions 分開，但同一個檔案 | 中型 spec |
| `'tags'` | 依 OpenAPI tag 拆分為多個檔案 | 大型 spec（推薦） |
| `'tags-split'` | tags + split 的組合 | 超大型 spec |

ModSemi 預設使用 `'tags'` 模式，每個 OpenAPI tag（如 `User`、`Article`）對應一個生成的 TypeScript 檔案。

#### `output.client` — 客戶端類型

| 值 | 說明 |
|----|------|
| `'axios'` | 工廠函式，支援注入自訂 instance（目前使用） |
| `'axios-functions'` | 純函式，直接使用 axios |
| `'react-query'` | 生成 `useQuery` / `useMutation` hooks |
| `'swr'` | 生成 SWR hooks |
| `'fetch'` | 使用原生 Fetch API |

#### `output.override.mutator` — 自訂 Instance

```typescript
override: {
  mutator: {
    path: './src/api/instance.ts',  // 指向 instance 檔案
    name: 'customInstance',         // 導出函式名稱
  },
},
```

### 14.6 生成結果解析

以 `swagger/sample.json` 中的 User tag 為例，生成的 `src/api/generated/user.ts` 結構如下：

```typescript
// ⚠️ 自動生成，請勿手動修改
import type {
  CreateUserRequest,
  GetUsersParams,
  UpdateUserRequest,
  User,
  UserListResponse,
} from './model';
import { customInstance } from '../instance';

// orval 使用工廠函式模式（client: 'axios'）
export const getUser = () => {
  const getUsers = (params?: GetUsersParams) =>
    customInstance<UserListResponse>({ url: `/users`, method: 'GET', params });

  const createUser = (createUserRequest: CreateUserRequest) =>
    customInstance<User>({
      url: `/users`,
      method: 'POST',
      headers: { 'Content-Type': 'application/json' },
      data: createUserRequest,
    });

  const getUserById = (id: number) =>
    customInstance<User>({ url: `/users/${id}`, method: 'GET' });

  const updateUser = (id: number, updateUserRequest: UpdateUserRequest) =>
    customInstance<User>({
      url: `/users/${id}`,
      method: 'PUT',
      data: updateUserRequest,
    });

  const deleteUser = (id: number) =>
    customInstance<void>({ url: `/users/${id}`, method: 'DELETE' });

  return { getUsers, createUser, getUserById, updateUser, deleteUser };
};
```

### 14.7 在組件中使用生成的 API

#### 基本使用方式

```typescript

import { getUser } from '../../api/generated/user';
import type { User } from '../../api/generated/model';

// 建立 API 實例（工廠函式模式）
const userApi = getUser();

export function UserList() {
  const [users, setUsers] = useState<User[]>([]);
  const [loading, setLoading] = useState(false);

  useEffect(() => {
    setLoading(true);
    userApi
      .getUsers({ page: 1, pageSize: 20 })
      .then(res => setUsers(res.data ?? []))
      .finally(() => setLoading(false));
  }, []);

  return (
    <div>
      {loading ? '載入中...' : users.map(u => <div key={u.id}>{u.name}</div>)}
    </div>
  );
}
```

#### 推薦封裝為自訂 Hook

```typescript
// src/hooks/useUsers.ts
import { useState, useEffect, useCallback } from 'react';
import { getUser } from '../api/generated/user';
import type { GetUsersParams, User, UserListResponse } from '../api/generated/model';

const userApi = getUser();

export function useUsers(params?: GetUsersParams) {
  const [data, setData] = useState<UserListResponse | null>(null);
  const [loading, setLoading] = useState(false);
  const [error, setError] = useState<Error | null>(null);

  const fetch = useCallback(async () => {
    setLoading(true);
    setError(null);
    try {
      const res = await userApi.getUsers(params);
      setData(res);
    } catch (e) {
      setError(e as Error);
    } finally {
      setLoading(false);
    }
  }, [params]);

  useEffect(() => { fetch(); }, [fetch]);

  return { data, loading, error, refetch: fetch };
}

// 在組件中使用
const { data, loading, refetch } = useUsers({ page: 1, pageSize: 20 });
```

### 14.8 多服務 / 多 Swagger 設定

當專案後端有多個 micro-service，各自提供不同的 Swagger spec，只需在 `orval.config.ts` 新增多個設定區塊：

```typescript
export default defineConfig({
  // 使用者服務
  userService: {
    input: { target: './swagger/user-service.json' },
    output: {
      mode: 'tags',
      target: './src/api/generated/user-service',
      schemas: './src/api/generated/user-service/model',
      client: 'axios',
      override: {
        mutator: { path: './src/api/instance.ts', name: 'customInstance' },
      },
    },
  },

  // 訂單服務
  orderService: {
    input: { target: './swagger/order-service.json' },
    output: {
      mode: 'tags',
      target: './src/api/generated/order-service',
      schemas: './src/api/generated/order-service/model',
      client: 'axios',
      override: {
        mutator: { path: './src/api/instance.ts', name: 'customInstance' },
      },
    },
  },
});
```

執行 `pnpm api:gen` 後，兩個服務的程式碼會分別生成到各自的目錄下：

```
src/api/generated/
├── user-service/
│   ├── user.ts
│   └── model/
└── order-service/
    ├── order.ts
    └── model/
```

### 14.10 環境變數與 baseURL 設定

#### 修改 API baseURL

`src/api/instance.ts` 預設 `baseURL` 為 `/api`（走 Modern.js 的 API proxy）。若需要指向不同環境的 API，透過環境變數控制：

**方式一：Modern.js 環境變數（推薦）**

在專案根目錄建立 `.env.local`（已加入 `.gitignore`）：

```bash
# .env.local
MODERN_APP_API_BASE_URL=https://api.your-backend.com
```

在 `modern.config.ts` 的 `source.define` 中注入（如尚未設定）：

```typescript
// modern.config.ts
export default defineConfig({
  source: {
    define: {
      'process.env.MODERN_APP_API_BASE_URL': JSON.stringify(
        process.env.MODERN_APP_API_BASE_URL ?? '/api'
      ),
    },
  },
});
```

**方式二：直接修改 instance.ts**

若不需要多環境設定，直接在 `src/api/instance.ts` 修改 `BASE_URL`：

```typescript
const BASE_URL = 'https://api.your-backend.com';
```

#### Modern.js API Proxy（本地開發推薦）

若後端在本地 `3001` 埠，可在 `modern.config.ts` 設定 proxy，讓 `/api` 自動轉發：

```typescript
export default defineConfig({
  dev: {
    server: {
      proxy: {
        '/api': {
          target: 'http://localhost:3001',
          changeOrigin: true,
          pathRewrite: { '^/api': '' },
        },
      },
    },
  },
});
```

### 14.11 常見問題 FAQ

**Q：執行 `pnpm api:gen` 後，出現 `Cannot find module 'orval'` 錯誤？**

A：確認已執行 `pnpm install`，且 `orval` 已列在 `devDependencies` 中。

---

**Q：生成的型別屬性都是 Optional（`?:`），但後端保證一定有值，如何處理？**

A：這是因為 OpenAPI spec 中未標記 `required`。有兩種解法：

1. **修改 spec**（推薦）：在 spec 的 `required` 陣列中宣告必填欄位
2. **使用 TypeScript 型別覆蓋**：在 `src/api/` 建立自訂型別，用 `Required<User>` 或 Utility Types 覆蓋

---

**Q：如何只重新生成某一個 API（不全部重生成）？**

A：目前 orval CLI 不支援單一 spec 重生成，只能使用 `pnpm api:gen` 全部重新生成。若 spec 較多，可考慮在 `orval.config.ts` 中暫時將其他 spec 設定區塊注解掉。

---

**Q：生成的程式碼有 lint 錯誤，要修嗎？**

A：**不需要**。`src/api/generated/` 已加入 `.gitignore`，且 biome 的 lint 設定不應套用到生成目錄。若 IDE 顯示 lint 錯誤，可在 `biome.json` 中忽略此目錄：

```json
{
  "files": {
    "ignore": ["src/api/generated/**"]
  }
}
```

---

**Q：如何讓生成的 API 函式支援取消請求（AbortController）？**

A：透過 `customInstance` 的 `AxiosRequestConfig` 傳入 `signal` 參數：

```typescript
const controller = new AbortController();

// 傳入 signal（生成函式通常接受額外的 config 參數）
userApi.getUsers({ page: 1 });

// 取消請求
controller.abort();
```

## 15. Dev Proxy — 解決本地開發 CORS 問題

### 15.1 為什麼需要 Proxy？

瀏覽器的**同源政策（Same-Origin Policy）** 規定，前端頁面只能對相同協定 + 域名 + 埠的伺服器發送 AJAX 請求。當前端跑在 `http://localhost:3000`、後端跑在 `http://localhost:3001` 時，兩者埠不同，屬於跨域，瀏覽器會阻擋回應並報錯：

```
Access to XMLHttpRequest at 'http://localhost:3001/users'
from origin 'http://localhost:3000' has been blocked by CORS policy
```

**解法**：讓 Modern.js Dev Server 充當中介，把前端對 `/api/**` 的請求在**伺服器端**轉發給後端。瀏覽器只看到同源的 `/api`，完全不知道跨域的存在。

```
瀏覽器（localhost:3000）
  │
  └─ GET /api/users          ← 同源，瀏覽器放行
        │
  Modern.js Dev Server
        │
        └─ GET /users        ← 伺服器端轉發（沒有 CORS 限制）
              │
         後端（localhost:3001）
```

---

### 15.2 設定方式

**檔案**：`modern.config.ts`

```typescript
export default defineConfig({
  // ...
  dev: {
    server: {
      proxy: {
        '/api': {
          target: 'http://localhost:3001',   // ← 後端位址，依實際情況修改
          changeOrigin: true,                // 修改 Host header，避免後端拒絕
          pathRewrite: { '^/api': '' },      // 移除 /api 前綴（見 15.3）
        },
      },
    },
  },
});
```

#### 各欄位說明

| 欄位 | 說明 |
|------|------|
| `'/api'` | 攔截所有以 `/api` 開頭的請求 |
| `target` | 轉發目標的後端位址 |
| `changeOrigin: true` | 將請求的 `Host` header 改為 `target` 的域名，避免後端因 Host 不符而拒絕 |
| `pathRewrite` | 轉發前對路徑做字串替換（見 15.3） |

#### 修改後端位址

只需改 `target` 即可，其他設定無需動：

```typescript
// 本地後端
target: 'http://localhost:3001'

// 遠端測試環境
target: 'https://staging-api.example.com'

// 遠端正式環境（不建議，應使用正式 CORS 設定）
target: 'https://api.example.com'
```

> **注意**：`dev.proxy` 設定只在 `pnpm dev` 時生效，執行 `pnpm build` 打包後不會包含 proxy 邏輯。

---

### 15.3 pathRewrite 使用時機

`pathRewrite` 決定轉發到後端時是否保留 `/api` 前綴，視後端 route 設計而定：

**情況一：後端 route 沒有 `/api` 前綴（需要 pathRewrite）**

```
後端 route：GET /users
前端呼叫：  GET /api/users

設定：pathRewrite: { '^/api': '' }
轉發結果：  GET /users  ✅
```

**情況二：後端 route 本身就有 `/api` 前綴（不需要 pathRewrite）**

```
後端 route：GET /api/users
前端呼叫：  GET /api/users

設定：刪除 pathRewrite 這行
轉發結果：  GET /api/users  ✅
```

對應 `modern.config.ts` 的修改：

```typescript
// 情況一（目前預設設定）
dev: {
  server: {
    proxy: {
      '/api': {
        target: 'http://localhost:3001',
        changeOrigin: true,
        pathRewrite: { '^/api': '' },  // 移除前綴
      },
    },
  },
},

// 情況二（後端 route 含 /api）
dev: {
  server: {
    proxy: {
      '/api': {
        target: 'http://localhost:3001',
        changeOrigin: true,
        // 不需要 pathRewrite
      },
    },
  },
},
```

### 15.4 多後端服務 Proxy

若專案有多個後端微服務，可設定多組 proxy 規則，各自指向不同服務：

```typescript
dev: {
  server: {
    proxy: {
      // 使用者服務
      '/api/users': {
        target: 'http://localhost:3001',
        changeOrigin: true,
        pathRewrite: { '^/api': '' },
      },

      // 訂單服務
      '/api/orders': {
        target: 'http://localhost:3002',
        changeOrigin: true,
        pathRewrite: { '^/api': '' },
      },

      // 檔案上傳服務
      '/api/upload': {
        target: 'http://localhost:3003',
        changeOrigin: true,
        pathRewrite: { '^/api': '' },
      },
    },
  },
},
```

> **比對順序**：Modern.js proxy 採用**最長前綴優先**原則，較具體的路徑（如 `/api/users`）會比通用路徑（如 `/api`）優先比對。

### 15.5 正式環境的 CORS 處理

`dev.proxy` 僅解決**本地開發**的 CORS 問題。部署到正式環境後，需擇一採用以下方案：

#### 方案 A：Nginx 反向代理（推薦）

讓前端與後端共用同一域名，由 Nginx 根據路徑分流：

```nginx
server {
    listen 80;
    server_name your-domain.com;

    # 前端靜態資源
    location / {
        root /var/www/modsemi/dist;
        try_files $uri $uri/ /index.html;
    }

    # API 請求轉發給後端（同域，無 CORS）
    location /api/ {
        proxy_pass http://backend:3001/;
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
    }
}
```

此方案前後端同源，瀏覽器不會觸發 CORS，是最穩健的做法。

#### 方案 B：後端設定 CORS Header

若前後端部署在不同域名，由後端在 HTTP 回應加上允許跨域的 header：

```
Access-Control-Allow-Origin: https://your-frontend.com
Access-Control-Allow-Methods: GET, POST, PUT, DELETE, OPTIONS
Access-Control-Allow-Headers: Content-Type, Authorization
```

各語言框架的設定方式：

| 語言 / 框架 | 套件 / 方式 |
|------------|-----------|
| Node.js / Express | `cors` 套件 |
| Node.js / NestJS | `@nestjs/platform-express` 內建 `enableCors()` |
| Python / FastAPI | `CORSMiddleware` |
| Java / Spring Boot | `@CrossOrigin` 或 `CorsConfiguration` |
| Go / Gin | `github.com/gin-contrib/cors` |

#### 方案 C：CDN / API Gateway

使用 AWS CloudFront、Cloudflare、Nginx 等在 CDN 層統一處理，前端與 API 使用同一個頂級域名（如 `api.example.com`），由 CDN 路由至不同後端。
